# 实时语音识别 — 处理管线详解

## 整体架构

```
浏览器麦克风                     服务器
┌──────────┐    WebSocket     ┌──────────────────────────────┐
│AudioWorklet├──── PCM 帧 ───►│ VAD → 语音缓冲 → Whisper 推理│
│ 16kHz mono│◄── JSON 结果 ───┤                              │
└──────────┘                  └──────────────────────────────┘
```

## 三层处理节奏

### 第一层：传输 — 每 100ms

前端 AudioWorklet 把麦克风音频攒到 1600 个采样点（16kHz × 0.1s = 1600），打包成 float32 ArrayBuffer 通过 WebSocket 发给服务器。

这是纯传输行为，不涉及任何识别判断。

### 第二层：VAD 分句 — 由静音时长触发

服务器收到 PCM 数据后，逐帧（每帧 30ms）送入 WebRTC VAD 判断是否有人在说话：

| VAD 状态 | 条件 | 行为 |
|---|---|---|
| IDLE → COLLECTING | 检测到语音帧 | 开始攒 buffer |
| COLLECTING 中 | 持续检测到语音 | 继续攒 buffer |
| COLLECTING 中 | 检测到静音但 < 600ms | 继续攒（保留短停顿） |
| COLLECTING → IDLE | 静音 ≥ 600ms | **commit**: 把整段 buffer 提交给 Whisper |

**为什么是 600ms？**

| 停顿场景 | 典型时长 | 是否触发 commit |
|---|---|---|
| 同一句话内的词间停顿 | 100-300ms | 否，保留在 buffer 中 |
| 逗号 / 分句处 | 300-600ms | 边缘，偶尔触发 |
| 句号 / 换气处 | 500-1000ms | 是，基本都触发 |
| 思考 / 话题切换 | 1000ms+ | 是 |

这个阈值保证了句内不误切、句间及时提交。通过 `.env` 的 `VAD_SILENCE_THRESHOLD_MS` 可调。

### 第三层：临时预览 — 每 300ms

在用户**正在说话**时（COLLECTING 状态），每 300ms 取当前 buffer 的快照做一次快速推理（beam_size=1），把中间结果发给前端显示为灰色斜体。

这样用户不用等说完才看到文字，说话过程中就能看到实时跟随的预览。

## 参数总结

| 参数 | 默认值 | 配置项 | 说明 |
|---|---|---|---|
| PCM chunk 大小 | 1600 samples (100ms) | 前端 AudioWorklet 硬编码 | 传输粒度 |
| VAD 帧长 | 30ms | `vad_frame_ms` | VAD 判断粒度 |
| VAD 激进度 | 2 | `VAD_AGGRESSIVENESS` (0-3) | 越大越激进过滤背景噪音 |
| 静音提交阈值 | 600ms | `VAD_SILENCE_THRESHOLD_MS` | 多长静音算一句话说完 |
| 最短语音片段 | 250ms | `VAD_MIN_SPEECH_MS` | 低于此的语音片段丢弃 |
| 最长单段音频 | 30s | 代码硬编码 | 超长语音强制提交 |
| Interim 间隔 | 300ms | 代码硬编码 | 临时预览的刷新频率 |
| 正式推理参数 | beam=5, best_of=5 | `beam_size`, `best_of` | 精度优先 |
| 预览推理参数 | beam=1, best_of=1 | 代码硬编码 | 速度优先 |

## 超时机制

| 超时类型 | 配置项 | 触发条件 | 行为 |
|---|---|---|---|
| 会话超时 | `SESSION_TIMEOUT_SEC` | 连接总时长超限 | 发 SESSION_TIMEOUT 错误后断开 |
| 静音超时 | `SILENCE_TIMEOUT_SEC` | 持续无语音超限（含连接后不说话） | 发 SILENCE_TIMEOUT 错误后断开 |
